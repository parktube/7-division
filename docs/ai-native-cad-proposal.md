# AI-Native CAD: 제안서

> **"AI가 만들고, AI가 사용한다"**
> — 인간은 의도를 전달하고, 결과를 검증한다

---

## 1. 왜 AI-Native인가?

### 기존 접근: AI-Assisted

```
인간 → [도구] → 결과
        ↑
       AI가 보조
```

- 인간이 도구를 "조작"
- AI는 자동완성, 추천, 오류 수정 정도
- 본질적으로 "인간이 배워야 하는 도구"

### 우리의 접근: AI-Native

```
인간 → [의도] → AI → [도구] → 결과
                      ↓
                 인간이 검증
```

- AI가 도구를 "조작"
- 인간은 의도를 전달하고 결과를 검증
- 본질적으로 "AI가 사용하는 도구"

---

## 2. 핵심 철학

### 철학 1: 도구의 사용자가 바뀌었다

```
AutoCAD 1982: 인간이 마우스로 조작
AutoCAD 2025: 여전히 인간이 마우스로 조작

AI-Native CAD: AI가 API로 조작
               인간은 의도 전달 + 검증
```

**함의**: UI/UX 설계의 대상이 "인간의 손"에서 "AI의 추론"으로 이동한다.

### 철학 2: 학습 곡선의 제거

```
기존 전문 도구:
- 6개월 학습 → 기본 사용
- 3년 경력 → 능숙한 사용
- "전문가"라는 희소성

AI-Native 도구:
- 0분 학습 → 의도 전달
- AI가 전문성 제공
- 인간은 도메인 지식에 집중
```

**함의**: "도구 사용법"은 AI가 학습하고, 인간은 "무엇을 만들 것인가"에 집중한다.

### 철학 3: 인터페이스의 역전

```
기존:
도구 → 인터페이스 → 인간
       (GUI/CLI)

AI-Native:
도구 → 인터페이스 → AI → 인터페이스 → 인간
       (API)              (자연어/시각)
```

**함의**: 도구-AI 인터페이스(API)와 AI-인간 인터페이스(자연어)를 분리 설계한다.

## 3. AX 가이드와의 연결

AX 설계 가이드의 핵심 원칙들이 AI-Native에서 어떻게 적용되는가:

### "LLM은 도구가 아니라 파트너"

```
AX 관점:
인간 ↔ LLM(파트너) ↔ 도구

AI-Native 확장:
인간 ↔ LLM(파트너) ↔ 도구
         ↑
    "파트너가 도구를 조작"
    "인간은 파트너와 대화"
```

### "추론을 막지 않는다"

```
AI-Native에서의 의미:
- 도구는 AI가 이해하기 쉬운 구조
- 도메인 6개 + 추론 (100개 명령어 X)
- ActionHints로 다음 행동 유도
```

### "협업은 자동화가 아니다"

```
AI-Native에서의 의미:
- AI가 "알아서 다 하는 것"이 아님
- 인간의 의도 확인 + 결과 검증 필수
- 피드백 루프가 핵심
```

## 4. 구현 방법론

### 4.1 두 개의 인터페이스 설계

```
┌─────────────────────────────────────────┐
│              AI-Native CAD              │
├─────────────────────────────────────────┤
│                                         │
│  [인간-AI 인터페이스]                    │
│  - 자연어 입력                          │
│  - 시각적 결과 확인                      │
│  - 승인/거부/피드백                      │
│                                         │
├─────────────────────────────────────────┤
│                                         │
│  [AI-도구 인터페이스]                    │
│  - 의미론적 API (시맨틱)                 │
│  - ActionHints                          │
│  - 진행상황 투명성                       │
│                                         │
├─────────────────────────────────────────┤
│                                         │
│  [코어 엔진]                            │
│  - 기하학 연산                          │
│  - 제약 솔버                            │
│  - 파일 I/O                             │
│                                         │
└─────────────────────────────────────────┘
```

### 4.2 API 설계 원칙: AI가 이해하기 쉽게

```
❌ 인간 중심 설계 (기존):
- 메뉴 계층 구조
- 단축키 조합
- 컨텍스트 메뉴
- 모달 다이얼로그

✅ AI 중심 설계:
- 평탄한(flat) 도메인 구조
- 이름만 봐도 의도가 명확한 함수
- 상태가 아닌 의도 기반 파라미터
- 결과에 다음 행동 힌트 포함
```

### 4.3 예시: 벽 그리기

```typescript
// 인간-AI 대화
인간: "4미터 x 3미터 방을 그려줘"

// AI의 추론 (내부)
// - 방 = 4개의 벽
// - 벽 = 폐합된 폴리라인
// - 치수 표기 필요할 수 있음

// AI-도구 호출
create_room({
  width: 4000,  // mm
  height: 3000,
  walls: { thickness: 150 }
})

// 도구 응답 (AI에게)
{
  result: { roomId: "r1", walls: ["w1", "w2", "w3", "w4"] },
  actionHints: {
    recommended: [
      { action: "add_door", reason: "방에 출입구 필요" },
      { action: "add_dimensions", reason: "치수 표기 권장" }
    ]
  }
}

// AI → 인간
AI: "4m x 3m 방을 그렸습니다. 문이나 창문을 추가할까요?"
    [미리보기 이미지]
    [승인] [수정 요청]
```

## 5. 개발 프로세스: AI와 함께 만들기

### 5.1 AI-Driven Development

```
기존 개발:
1. 요구사항 정의 (인간)
2. 설계 (인간)
3. 구현 (인간 + AI 보조)
4. 테스트 (인간)

AI-Native 개발:
1. 의도 공유 (인간 ↔ AI)
2. 설계 제안 (AI) → 검증 (인간)
3. 구현 (AI) → 검증 (인간)
4. AI가 사용해보기 (AI) → 피드백 (AI + 인간)
```

### 5.2 "AI가 사용해보기" 단계의 중요성

```
새 API를 만들면:
1. AI에게 실제로 사용하게 함
2. AI가 사용하기 어려운 부분 발견
3. AI의 피드백으로 API 개선
4. 반복

"인간이 사용하기 편한가?" → "AI가 추론하기 쉬운가?"
```

### 5.3 문서화의 변화

```
기존 문서:
- 튜토리얼 10페이지
- 레퍼런스 100페이지
- 예제 50개

AI-Native 문서:
- 도메인 설명 1페이지
- 각 도구의 시맨틱 1줄
- ActionHints가 문서 역할

"AI가 읽고 이해할 수 있으면 된다"
```

## 6. 검증 루프 설계

### 6.1 AI가 할 수 없는 것

```
아무리 AI-Native여도:
1. 인간의 의도를 대신 정할 수 없음
2. "좋다/나쁘다"를 대신 판단할 수 없음
3. 도메인 지식의 최종 결정권 없음

따라서:
- 의도 확인 → 필수
- 결과 검증 → 필수
- 피드백 수용 → 필수
```

### 6.2 최소 검증 UI

```
필수 요소:
┌─────────────────────────────────┐
│  [실시간 뷰어]                   │
│  - 현재 상태 시각화              │
│  - 변경사항 하이라이트           │
│                                 │
│  [컨트롤]                        │
│  - ✓ 승인  ✗ 거부               │
│  - 💬 코멘트                     │
│  - ↩ 되돌리기                   │
│  - ▶ 재실행                     │
│                                 │
│  [진행상황]                      │
│  - 현재 단계                     │
│  - 다음 예정 행동                │
│  - 근거/로그 링크                │
└─────────────────────────────────┘
```

## 8. 제안: 첫 번째 마일스톤

### Phase 0: 철학 정립 (이 문서)
- AI-Native 접근의 합의
- AX 가이드와의 정합성 확인

### Phase 1: 최소 검증 — "사람 스켈레톤을 그려줘"

**목표**: 기초 도형 도구만으로 AI가 복잡한 형상을 만들 수 있는지 검증

**제공하는 도구 (기초 도형만)**:
```typescript
// 도메인 지식은 도구에 넣지 않는다
// AI가 추론해서 조합한다

line(x1, y1, x2, y2)      // 선
circle(x, y, radius)       // 원
rect(x, y, width, height)  // 사각형
move(id, dx, dy)           // 이동
rotate(id, angle, cx, cy)  // 회전
delete(id)                 // 삭제
```

**검증 시나리오**:
```
인간: "사람 스켈레톤을 그려줘"

AI 추론 (도메인 지식은 AI가 가짐):
- 머리 = 원
- 척추/팔/다리 = 선
- 관절 연결 구조 이해

AI 도구 호출:
1. circle(0, 100, 10)       // 머리
2. line(0, 90, 0, 50)       // 척추
3. line(0, 85, -20, 70)     // 왼쪽 상완
4. line(-20, 70, -25, 50)   // 왼쪽 하완
5. line(0, 85, 20, 70)      // 오른쪽 상완
6. line(20, 70, 25, 50)     // 오른쪽 하완
7. line(0, 50, -15, 20)     // 왼쪽 대퇴
8. line(-15, 20, -15, 0)    // 왼쪽 하퇴
9. line(0, 50, 15, 20)      // 오른쪽 대퇴
10. line(15, 20, 15, 0)     // 오른쪽 하퇴

인간: [결과 확인] "팔을 더 길게 해줘"

AI: move/delete/재생성으로 수정
```

**검증 기준**:
- [ ] AI가 "스켈레톤"을 기초 도형으로 분해할 수 있는가?
- [ ] 도구 호출 순서가 합리적인가?
- [ ] 수정 요청에 적절히 대응하는가?
- [ ] 도구 API가 AI에게 직관적인가?

**왜 이 접근인가**:
```
❌ create_skeleton({ type: "humanoid" })
   → 도메인 지식을 "도구에" 넣음
   → AI는 그냥 호출만 함
   → AI의 추론 능력을 활용 못함

✅ line, circle, rect + AI 추론
   → 도메인 지식을 "AI가" 가짐
   → AI가 추론해서 조합
   → 진짜 "AI가 도구를 사용하는" 테스트
```

### Phase 2: 도메인 확장
- Phase 1 도구 + 그룹화, 레이어
- 더 복잡한 요청: "팔을 구부린 포즈로", "걷는 자세로"
- ActionHints 동작 확인
- 검증 UI 프로토타입

### Phase 3: 실사용 테스트
- 실제 작업 시도 (캐릭터 리깅용 스켈레톤 등)
- 인간-AI 협업 플로우 검증
- 피드백 → 개선 반복
